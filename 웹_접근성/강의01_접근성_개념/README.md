# 강의01 — 웹 접근성 개념

## 학습 목표

- 웹 접근성의 정의와 법적 의무를 이해한다
- 장애 유형별 보조기술을 파악한다
- WCAG 2.1의 4원칙을 설명할 수 있다
- Lighthouse와 axe DevTools로 접근성을 측정할 수 있다

---

## 1. 웹 접근성이란?

**웹 접근성(Web Accessibility):** 장애가 있는 사람, 고령자 등 **모든 사람**이 웹 콘텐츠에 접근하고 사용할 수 있도록 보장하는 것

> "웹의 힘은 보편성에 있다. 장애에 관계없이 모든 사람이 접근할 수 있는 것이 웹의 핵심이다." — 팀 버너스리 (WWW 창시자)

### 왜 필요한가?

1. **법적 의무:** 공공기관 및 일정 규모 이상의 민간 웹사이트에 접근성 준수 의무
2. **더 많은 사용자:** 국내 장애인 인구 약 260만명, 고령자 포함 시 훨씬 많음
3. **SEO 향상:** 접근성 개선은 검색엔진 최적화에도 도움
4. **더 나은 UX:** 접근성 개선은 비장애인에게도 편의성 향상

### 한국 법적 근거

- **장애인차별금지법 제21조:** 웹사이트 접근성 보장 의무
- **한국형 웹 콘텐츠 접근성 지침(KWCAG) 2.1:** 준수 기준
- 위반 시 시정 명령 및 과태료 부과 가능

## 2. 장애 유형별 보조기술

| 장애 유형 | 보조기술 | 웹에서의 접근 방식 |
|-----------|----------|-------------------|
| 시각 장애 | 스크린리더 (NVDA, VoiceOver, JAWS) | HTML 구조와 텍스트 대안에 의존 |
| 저시력 | 화면 확대 소프트웨어, 고대비 모드 | 텍스트 크기, 색상 대비에 민감 |
| 청각 장애 | 자막, 수화 영상 | 멀티미디어 대안 콘텐츠 필요 |
| 운동 장애 | 키보드만 사용, 스위치 제어 | 마우스 없이 모든 기능 사용 가능해야 |
| 인지 장애 | 다양한 보조 앱 | 단순하고 명확한 UI 필요 |

### 스크린리더 동작 방식

```html
<!-- 스크린리더는 이렇게 읽음 -->
<nav>
    <ul>
        <li><a href="/">홈</a></li>      <!-- "홈 링크" 라고 읽음 -->
        <li><a href="/about">소개</a></li>
    </ul>
</nav>

<button disabled>제출</button>           <!-- "제출 버튼 비활성화됨" 라고 읽음 -->

<img src="logo.png" alt="회사 로고">    <!-- "회사 로고 이미지" 라고 읽음 -->
<img src="deco.png" alt="">             <!-- 장식 이미지는 건너뜀 -->
```

## 3. WCAG 2.1 4원칙

**WCAG (Web Content Accessibility Guidelines):** W3C가 만든 국제 웹 접근성 표준

### 원칙 1. 인식 가능 (Perceivable)

정보가 모든 사용자에게 인식 가능해야 합니다.

```html
<!-- 텍스트 대안 제공 -->
<img src="chart.png" alt="2024년 1분기 매출: 1억 2천만원">

<!-- 시간 기반 미디어 대안 -->
<video>
    <track kind="captions" src="subtitles.vtt" srclang="ko">
</video>

<!-- 색상만으로 정보 전달 금지 -->
<!-- 나쁜 예: 빨간색 = 오류 (색맹 사용자는 알 수 없음) -->
<!-- 좋은 예: 빨간색 + 오류 아이콘 + 텍스트 메시지 -->
```

### 원칙 2. 운용 가능 (Operable)

인터페이스를 모든 방식으로 조작할 수 있어야 합니다.

```html
<!-- 모든 기능이 키보드로 사용 가능해야 함 -->
<button onclick="submitForm()">제출</button>  <!-- 좋음: 키보드로 접근 가능 -->
<div onclick="submitForm()">제출</div>        <!-- 나쁨: 키보드로 접근 불가 -->

<!-- 충분한 시간 제공 -->
<!-- 자동 슬라이드는 정지/일시정지 버튼 제공 -->
```

### 원칙 3. 이해 가능 (Understandable)

콘텐츠와 조작이 이해 가능해야 합니다.

```html
<!-- 페이지 언어 선언 -->
<html lang="ko">

<!-- 에러 메시지는 구체적으로 -->
<!-- 나쁜 예: "입력 오류" -->
<!-- 좋은 예: "이메일 형식이 올바르지 않습니다. (예: user@example.com)" -->
```

### 원칙 4. 견고 (Robust)

다양한 보조기술이 콘텐츠를 해석할 수 있어야 합니다.

```html
<!-- 올바른 HTML 작성 -->
<!-- 여는 태그에는 닫는 태그 -->
<!-- 중첩 구조 올바르게 -->
<!-- ARIA 속성 올바르게 사용 -->
```

### 적합성 수준

| 수준 | 설명 |
|------|------|
| A | 최소 요구사항 (미충족 시 일부 사용자 완전 차단) |
| AA | 권장 수준 (대부분 법적 기준이 AA) |
| AAA | 최고 수준 (모든 상황에서 완전 충족) |

## 4. 접근성 테스트 도구

### Lighthouse (Chrome DevTools)

1. Chrome에서 페이지 열기
2. F12 → `Lighthouse` 탭
3. `Accessibility` 체크 → `Analyze page load`
4. 점수 및 개선 항목 확인

### axe DevTools

1. Chrome 웹스토어에서 **axe DevTools** 설치
2. F12 → `axe DevTools` 탭 → `Scan ALL of my page`
3. 위반 항목과 수정 방법 확인

### 키보드 테스트

```
Tab        - 다음 포커스 가능 요소로 이동
Shift+Tab  - 이전 요소로 이동
Enter      - 링크 활성화, 버튼 클릭
Space      - 버튼/체크박스 토글
화살표키   - 라디오버튼, 선택 메뉴 탐색
Escape     - 모달/팝업 닫기
```
