# 5가지 실수 상황별 복구 명령어

## 실수 1: 파일을 git add했는데 취소하고 싶다

### 상황

```bash
# 실수로 민감한 파일(secret.env)을 스테이징함
git add secret.env
git status
# Changes to be committed:
#   new file: secret.env  ← 이걸 취소하고 싶음
```

### 복구

```bash
# 방법 1: git restore --staged (Git 2.23+)
git restore --staged secret.env
# secret.env가 unstaged 상태로 돌아옴

# 방법 2: git reset HEAD (구버전 방식)
git reset HEAD secret.env

# 확인
git status
# Untracked files:
#   secret.env   ← 다시 untracked 상태
```

## 실수 2: 커밋 메시지를 잘못 썼다 (아직 push 전)

### 상황

```bash
git commit -m "feaaaat: 오타난 커밋 메시지"
git log --oneline
# abc1234 feaaaat: 오타난 커밋 메시지  ← 고쳐야 함
```

### 복구

```bash
# 방법 1: git commit --amend (가장 최근 커밋만)
git commit --amend -m "feat: 올바른 커밋 메시지"

# 방법 2: git reset --soft + 재커밋
git reset --soft HEAD~1     # 커밋 취소, 변경사항 스테이징 유지
git commit -m "feat: 올바른 커밋 메시지"

# 확인
git log --oneline
# def5678 feat: 올바른 커밋 메시지  ← 수정됨
```

> ⚠️ push된 커밋의 메시지는 `amend` 후 `--force push`가 필요합니다.  
> 팀 저장소에서는 팀원에게 알리고 진행하세요.

## 실수 3: 잘못 커밋한 파일이 있다 (아직 push 전)

### 상황

```bash
# .env 파일을 실수로 커밋함
git add .env
git commit -m "feat: 환경변수 추가"

git log --oneline
# abc1234 feat: 환경변수 추가  ← .env가 포함된 커밋 취소하고 싶음
```

### 복구

```bash
# 방법 1: 커밋은 취소하고 변경사항 유지 (git reset --mixed)
git reset HEAD~1        # 커밋 취소, .env가 unstaged 상태로 복귀
git status              # .env가 untracked로 보임

# .env를 .gitignore에 추가
echo ".env" >> .gitignore
git add .gitignore
git commit -m "chore: .env를 .gitignore에 추가"

# 방법 2: 특정 파일만 커밋에서 제거 (커밋 유지)
git reset HEAD~1 -- .env      # .env만 unstaged로 되돌림
git commit --amend --no-edit  # 커밋 재작성 (.env 제외됨)
```

## 실수 4: git reset --hard로 커밋을 날려버렸다

### 상황

```bash
# 중요한 커밋을 실수로 reset --hard로 삭제함
git log --oneline
# abc1234 feat: 중요한 기능 추가  ← 이게 사라짐!
# def5678 init: 초기화

git reset --hard HEAD~1
# HEAD is now at def5678 init: 초기화

git log --oneline
# def5678 init: 초기화   ← abc1234가 사라짐!
```

### 복구 (git reflog 사용!)

```bash
# reflog로 이동 이력 확인
git reflog
# def5678 HEAD@{0}: reset: moving to HEAD~1  ← 실수로 한 reset
# abc1234 HEAD@{1}: commit: feat: 중요한 기능 추가  ← 이 시점으로!
# def5678 HEAD@{2}: commit (initial): init: 초기화

# reset --hard를 reset --hard로 되돌리기
git reset --hard HEAD@{1}
# HEAD is now at abc1234 feat: 중요한 기능 추가

# 또는 해시 직접 사용
git reset --hard abc1234

# 확인
git log --oneline
# abc1234 feat: 중요한 기능 추가  ← 복구됨!
# def5678 init: 초기화
```

## 실수 5: 파일을 실수로 삭제하거나 망쳤다

### 상황 A: 커밋되지 않은 파일 변경사항을 되돌리고 싶다

```bash
# 실수로 중요한 파일을 수정해버림
echo "이상한 내용" >> important.js

# 마지막 커밋 시점으로 되돌리기
git restore important.js    # Git 2.23+
# 또는
git checkout -- important.js  # 구버전 방식

# 확인: 변경사항이 사라짐
git status
# nothing to commit
```

### 상황 B: push된 커밋을 되돌려야 한다

```bash
# 이미 push된 커밋에 버그가 있어서 되돌려야 함
git log --oneline
# abc1234 feat: 버그 있는 기능  ← 이걸 되돌리고 싶음
# def5678 feat: 이전 기능

# git revert — 새 "되돌리기 커밋" 추가 (히스토리 유지)
git revert abc1234
# [main xyz9999] Revert "feat: 버그 있는 기능"

git log --oneline
# xyz9999 Revert "feat: 버그 있는 기능"  ← 되돌리는 커밋
# abc1234 feat: 버그 있는 기능           ← 이전 커밋 보존
# def5678 feat: 이전 기능

# push
git push origin main
```

## 전체 복구 명령어 요약

| 상황 | 명령어 |
|------|--------|
| git add 취소 | `git restore --staged <파일>` |
| 마지막 커밋 메시지 수정 (push 전) | `git commit --amend -m "새 메시지"` |
| 마지막 커밋 취소 (변경사항 유지) | `git reset --soft HEAD~1` |
| 마지막 커밋 취소 (변경사항도 제거) | `git reset --hard HEAD~1` |
| 파일 변경사항 되돌리기 | `git restore <파일>` |
| 날린 커밋 복구 | `git reflog` → `git reset --hard HEAD@{N}` |
| push된 커밋 되돌리기 | `git revert <커밋해시>` |

## 긴급 상황에서 기억할 것

> 뭔가 잘못됐다는 느낌이 들면 바로 멈추고 `git status`와 `git log`로 현재 상태를 파악하세요.
> `git reflog`는 거의 모든 실수를 복구해줍니다.
